# P4 实验一：AI 代码信任度实验

## 1. 实验对象

功能点选择：代取状态更新中的“发布方确认完成”逻辑。

选择原因：

- 该逻辑涉及状态约束
- 涉及是否存在完成凭证
- 涉及有偿服务的支付结算分支
- 规模适中，适合比较 AI 直出与人工审查修复

## 2. Prompt

```text
请为 CampusHub 的 PickupServiceImpl 生成“发布方确认完成代取订单”的核心逻辑。
要求：
1. 只允许发布方确认完成；
2. 订单必须已经被接单且处于进行中；
3. 必须已经上传完成凭证；
4. 确认完成后状态改为 COMPLETED；
5. 若为有偿订单，需要调用支付服务进行结算；
6. 返回包含状态、支付状态、完成时间的结果对象；
7. 统一通过 BusinessException 返回业务错误。
```

## 3. AI 直出结果摘要

AI 首版给出的思路大体正确，能产出以下主干：

- 读取订单
- 判断当前用户是否为发布方
- 状态改为 `COMPLETED`
- 有偿订单调用支付结算
- 返回完成结果对象

但首版代码仍存在多处不能直接信任的问题。

## 4. 人工检查发现的问题

### 问题 1：缺少完成凭证校验

AI 直出版本只检查了状态是否为 `IN_PROGRESS`，未检查 `completionProofFileId` 是否已上传。  
风险：发布方可在接单方未提交完成凭证时直接确认完成，破坏业务流程。

### 问题 2：支付分支边界不完整

AI 直出版本默认“有 paymentId 就结算”，但没有结合 `rewardType` 判断。  
风险：无偿订单若出现脏数据，可能错误触发支付服务。

### 问题 3：异常信息偏泛化

AI 直出版本错误统一写成“操作非法”或“状态错误”。  
风险：不利于联调、测试和 Bug 复现，也不利于保持统一错误语义。

## 5. 修复方式

人工审查后补充了以下约束：

- 必须校验 `completionProofFileId != null`
- 只有 `RewardType.PAID` 且存在 `paymentId` 时才进行结算
- 使用统一的 `BusinessException + ErrorCode.CONFLICT/FORBIDDEN`
- 返回值与 `CompletionConfirmResult` 契约对齐

## 6. 修复前后对比

| 指标 | AI 直出 | 人工审查修复后 |
|------|--------|---------------|
| 编译是否通过 | 可以通过 | 可以通过 |
| 功能是否可运行 | 可运行，但存在业务漏洞 | 可运行 |
| 是否符合 P3 设计 | 部分符合 | 符合 |
| 测试是否能覆盖核心规则 | 不能完全覆盖 | 可以覆盖 |
| 主要问题 | 漏掉完成凭证校验、支付边界不严、异常语义粗糙 | 已补齐 |

## 7. 结论

本次实验说明：

- AI 在“生成标准业务骨架”方面效率很高
- 但涉及状态机、异常分支、支付边界时，AI 直出代码不能直接信任
- 人工审查的价值主要体现在：补边界、校契约、查状态流转、查异常语义

因此本团队在 P4 的结论是：  
AI 适合先产出业务主干，但核心流程代码必须经过人工审查后再合入。
