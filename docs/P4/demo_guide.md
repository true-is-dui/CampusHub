# P4 演示说明

## 1. 运行方式

项目在 `deploy/` 下提供三套环境配置，各自带 `docker-compose.yml`、`.env.example` 和 MySQL 初始化脚本（建表 + 演示数据）：

| 环境 | 目录 | 用途 | 组成 |
|------|------|------|------|
| dev | `deploy/dev/` | 团队本地开发 | 仅 MySQL 容器；后端/前端在本机用 `mvnw` / `npm run dev` 跑 |
| test | `deploy/test/` | 本地一键演示 | MySQL + 后端 + 前端 + Nginx 全容器化 |
| prod | `deploy/prod/` | 远程部署 | 与 test 同构，已部署到服务器 |

### 1.1 远程演示（推荐，无需本地环境）

已部署地址：**https://campushub.trueisdui.online**

直接在浏览器打开即可，数据来自 prod 初始化脚本（见第 2 节演示账号）。

### 1.2 本地一键演示（test 环境）

test 环境为全容器化，一条命令拉起 MySQL、后端、前端和 Nginx：

```bash
cd deploy/test
cp .env.example .env      # 按需修改数据库密码、JWT_SECRET 等
docker compose up -d --build
```

启动后：

- 访问地址：`http://localhost`（Nginx 监听 80，前端 `/`、后端 `/api/` 反代）
- MySQL 容器首次启动自动执行 `mysql/init/01-schema.sql`（建表）和 `02-data-test.sql`（演示数据）
- 后端文件存储、JWT 密钥等由 `.env` 注入

> `.env.example` 是模板，真实 `.env` 含密码，已被 `.gitignore` 忽略，请勿提交。

### 1.3 本地开发（dev 环境，供参考）

开发时只用容器起 MySQL，应用在本机跑，便于热重载调试：

```bash
cd deploy/dev && cp .env.example .env && docker compose up -d   # 仅 MySQL
cd ../../backend && ./mvnw spring-boot:run                       # 后端 :8080，基址 /api/v1
cd ../frontend && npm install && npm run dev                     # 前端 :5173（Vite proxy 转发 /api）
```

演示数据由初始化脚本（test 为 `deploy/test/mysql/init/02-data-test.sql`，prod 同构）自动灌入，**可重复重置**（重建容器即恢复）。固定账号如下（均为非敏感测试信息）：

| 账号 | 密码 | 角色 | 认证状态 | 说明 |
|------|------|------|---------|------|
| `admin` | `Admin@123` | 管理员 | 已通过 | 用于演示实名认证审核 |
| `alice` | `User@123` | 普通用户 | 已通过 | 已有 100 积分，可发布有偿代取 |
| `bob` | `User@123` | 普通用户 | 已通过 | 已有 100 积分，可接单 |
| `carol` | `User@123` | 普通用户 | 审核中 | 用于演示管理员审核通过/驳回 |
| `demo_user_01` ~ `demo_user_60` | `User@123` | 普通用户 | 多为已通过 | 批量陪衬数据 |

预置业务数据覆盖演示所需的各类状态：

- 代取请求约 159 条，覆盖 `WAITING_ACCEPT` / `IN_PROGRESS` / `COMPLETED` / `CANCELLED` 四种状态和 `PAID` / `UNPAID` 两种报酬类型
- 评价约 70 条（可展示好评率）、站内通知 200+ 条（可展示未读/已读）
- 已认证用户各带一条「认证赠送 100 积分」流水，积分余额与流水一致

若助教希望现场从头演示，可直接新建两名普通用户走完整流程（见路径 A）。

## 3. 核心演示路径

### 路径 A：完整主流程（含积分）

1. 注册发布方账号并登录（或直接用 `alice`）
2. 完善个人资料，提交实名认证（学号、姓名、认证材料）
3. 用 `admin` 登录，审核通过该实名认证 —— **认证通过系统赠送 100 积分**
4. （可选）发布方每日签到 —— **签到赠送 5 积分**，演示积分获取渠道
5. 发布代取需求并上传取件凭证：
   - 无报酬需求直接进入待接单
   - **有报酬需求发布时从发布方积分余额中扣减对应积分**后进入待接单（余额不足会被拒绝，见路径 B）
6. 用另一账号（如 `bob`）登录并接单，需求进入进行中
7. 接单方上传完成凭证
8. 发布方确认完成，需求进入已完成 —— **有报酬需求的积分转入接单方账户**
9. 双方互评，展示好评率
10. 在「我的发布 / 我的接单」中查看状态变化，在通知中心查看接单/完成/评价通知

### 路径 B：异常与取消流程展示

1. 未登录直接访问发布或接单接口，展示 `40101`
2. 有报酬需求缺少金额提交，展示 `40001`
3. 对同一需求重复接单，展示 `40901`
4. **积分不足时发布有报酬需求**，展示 `40901`（`INSUFFICIENT_POINTS`）
5. 发布方在待接单阶段主动取消需求，需求进入已取消；**有报酬需求取消时积分退回发布方**

## 4. 演示重点

- JWT 登录态是否正确传递
- 实名认证前后操作权限是否正确；管理员审核链路
- 代取状态是否按 `WAITING_ACCEPT -> IN_PROGRESS -> COMPLETED` 正常流转，以及 `CANCELLED`（主动取消 / 接单超时）终态
- 平台积分闭环：认证赠送、每日签到、发布扣减、取消退回、完成转移，积分变动与代取状态一致
- 上传图片和受保护文件访问是否正常
- 评价与好评率、站内通知未读/已读
- 异常提示是否清晰（40101 / 40001 / 40901）

## 5. 说明

本次 P4 演示以课程 MVP 为边界，不展示以下扩展内容：

- 失物招领
- 二手交易
- 私聊
- WebSocket
- Redis / MQ / 搜索优化
- 复杂管理后台
- 真实线上支付 / 支付宝沙箱（有偿代取以平台内虚拟积分计价）

这样可以保证演示链路聚焦且稳定。
