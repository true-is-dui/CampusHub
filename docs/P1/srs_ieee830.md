## 1. 引言

### 1.1 目的

本文档旨在定义“校园互助服务平台”的完整功能性和非功能性需求。该平台旨在连接校园内的学生，使他们能够发布和响应各类互助需求，构建一个安全、高效、可信的校园服务生态系统。本文档是项目团队进行设计、实现和测试的主要依据。

### 1.2 文档约定

- **优先级定义（MoSCoW）：**

  - `[Should]`：应该实现，除非有明确理由
  - `[Could]`：可以实现，视资源与时间而定
  - `[Won't]`：本次迭代不实现

  - 对于 `[AI生成+人工修改]` 的需求，在需求描述后以 `（修改说明：...）` 注明修改内容

### 1.3 预期的读者和阅读建议

- **开发/测试人员：** 重点关注第 3 章（功能需求）和第 4 章（非功能需求）

### 1.4 产品的范围

该产品是一个面向电脑端浏览器的校园互助 Web 平台。核心范围包括：用户注册与认证、校园代取服务、互助信息发布与响应、站内即时通讯、评价与好评率展示、后台管理功能。产品 MVP 收敛为两大板块：服务板块仅支持代取业务；互助板块支持失物招领和二手交易。找搭子、咨询问答、代购/代办等候选需求保留为后续扩展方向。

## 2. 总体描述

### 2.1 产品视角

该产品是一个独立运行的软件系统，初期不接入真实第三方支付生产环境，仅接入支付宝沙箱环境。用户通过电脑端浏览器访问。

### 2.2 产品功能（高层概要）

1.  **用户管理：** 注册、登录、学号实名认证、个人资料
2.  **需求大厅：** 按服务/互助板块浏览内容，支持当前板块内校区筛选、关键词搜索和发布时间倒序展示
3.  **服务板块：** 代取服务，覆盖快递、外卖和已明确物品的代取/带回
4.  **互助板块：** 失物招领、二手交易，采用轻量信息发布与私聊联系模式
5.  **内容与服务流：** 服务板块支持接单、完成与评价闭环；互助板块支持发布、审核、展示、私聊联系和发布者标记结束
6.  **评价体系：** 代取服务双方互评，好评率展示
7.  **即时通讯：** 代取服务、失物招领、二手交易场景内私聊
8.  **后台管理：** 用户封禁、内容审核、举报/申诉处理

### 2.3 用户特征

| 用户角色 | 特征描述 | 使用频率 | 技术水平 |
|---------|---------|---------|---------|
| **普通用户（学生）** | 所有注册并通过学号认证的在校大学生。**同一用户在不同服务中可同时作为需求方和接单方**，两种身份由具体服务上下文决定，非互斥。 | 高（每周 1-3 次） | 中 |
| **管理员** | 运营方人员，负责平台秩序管理、内容审核、用户封禁、申诉处理。 | 中 | 高 |

> **角色说明：** 平台不将“需求方”和“接单方”预设为用户类型，而是作为用户在具体服务中的**行为状态**。任何认证用户均可发布代取服务（此时为需求方），也可接单（此时为接单方），同一用户可同时持有多个不同状态的服务。

### 2.4 运行环境

- **客户端：** Chrome、Edge、Safari 等主流桌面浏览器
- **服务器端：** Linux 环境，Node.js/Python/Java，MySQL，Redis

### 2.5 设计和实现上的限制

- **学号唯一性：** 每个学号只能绑定一个账号，已认证学号不可重复使用 [人工补充]
- **账号注册：** MVP阶段使用用户名和密码注册登录，不接入运营商短信验证码；图形验证码可作为后续防机器人安全增强，不进入当前Must范围 [人工修改]
- **支付方式：** 有报酬代取服务的服务报酬通过支付宝沙箱进行预付款、转账和退款，MVP阶段不涉及真实资金；互助板块不接入平台支付 [AI生成+人工修改]（修改说明：原人工修改废除积分体系并接入支付宝沙箱，后续人工确认仅用于有报酬代取服务的沙箱预付款）
- **服务范围：** MVP 阶段服务板块仅支持代取业务，不支持代购、代办或任何需要接单方垫付本金的服务 [人工修改]
- **认证限制：** 用户参与平台活动前必须提交学号、姓名和校园卡/学生证照片，由管理员人工审核完成实名认证 [AI生成+人工修改]（修改说明：AI 原要求身份认证，人工明确认证材料、必选照片和管理员审核）
  - *修改说明：AI 原要求对接官方 API，改为管理员人工审核，降低初期对接成本*
- **隐私合规：** 学号不公开显示；手机号仅在用户主动填写并同意公开的场景展示 [人工补充]
- **开发周期：** 10 周内完成 MVP [人工补充]

### 2.6 假设和依赖

- **假设：** 用户可使用电脑访问校园网络或互联网；校园网络覆盖良好
- **依赖：** MVP 阶段不依赖学校身份验证接口，后续如学校提供统一认证接口可评估接入；依赖主流浏览器环境

#### 代取服务状态说明

> 来源：[AI生成+人工修改]（说明：AI 梳理服务状态，人工确认审核、接单、完成、取消和预付款相关边界；P4回溯后术语收敛为代取服务）

| 状态 | 含义 | 主要触发操作 |
|------|------|--------------|
| 待支付 | 有报酬代取服务已提交并创建支付宝沙箱预付款入口，等待发布者支付；仅发布者本人可见，不进入服务大厅和管理员内容审核队列 | 发布有报酬服务 |
| 审核中 | 代取服务等待内容审核；有报酬服务支付成功后进入，无报酬服务提交后直接进入 | 支付成功 / 发布无报酬服务 |
| 待接单 | 审核通过并公开展示，等待其他用户接单 | 审核通过 |
| 进行中 | 已有接单方承接服务，其他用户不可再接单 | 用户接单 |
| 已完成 | 发布方确认服务完成 | 发布方确认完成 |
| 已取消 | 服务被发布者取消、支付超时、审核驳回或接单截止超时取消，不再公开展示 | 发布者取消 / 支付超时 / 审核驳回 / 接单截止超时 |

#### 代取服务取消规则

> 来源：[AI生成+人工修改]（说明：AI 整理取消与退款规则，人工确认预付款退还、完成后不可撤回和接单方取消接单边界）

- 待支付状态下，发布方可继续支付或手动取消；超过支付有效期仍未支付时，系统自动取消服务。
- 待支付状态的服务仅发布者本人可见，不进入服务大厅和管理员内容审核队列。
- 审核中或待接单状态下，发布方可取消服务，服务状态变为已取消。
- 待接单状态下，服务超过接单截止时间仍无人接单时，系统自动取消服务。
- 进行中状态下，发布方不可单方面取消服务；接单方可取消接单，服务退回待接单，其他用户可重新接单。
- 接单方取消接单后，发布方可继续等待其他用户接单，也可在待接单状态下取消服务。
- 服务取消时，已完成预付款的有报酬服务按 6.2 支付系统说明通过支付宝沙箱原路退还预付款；待支付状态取消或超时取消时尚未完成预付款，不涉及退款；无报酬服务不涉及退款。
- 已完成服务不可撤回，不支持退款；如有争议，可通过评价举报/申诉或代取服务异常申诉处理。
- 代取服务发布后不支持直接修改；如内容有误，发布方可在审核中或待接单状态下取消后重新发布。
- 进行中服务异常情况按“进行中服务异常处理规则”处理。

#### 进行中服务异常处理规则

> 来源：[AI生成+人工修改]（说明：AI 根据接单方不完成、发布方不确认等问题整理异常处理规则，人工确认通过申诉由管理员处理）

- 进行中服务不做自动取消或自动完成。
- 若接单方长时间未完成服务且未主动取消接单，发布方可提交代取服务异常申诉。
- 若接单方已上传完成凭证但发布方长时间未确认完成，接单方可提交代取服务异常申诉。
- 管理员处理代取服务异常申诉时，可查看服务信息、完成凭证和聊天记录。
- 管理员可根据申诉处理结果将服务退回待接单、标记为已取消或标记为已完成。
- 有报酬服务被标记为已取消时，系统通过支付宝沙箱退还预付款；有报酬服务被标记为已完成时，系统通过支付宝沙箱将预付款转账至接单方。

### 3.3 互助板块说明

> 来源：[人工修改]（说明：P4回溯评审后，将互助板块收敛为失物招领和二手交易两类轻量互助信息帖，不引入论坛型问答或报名型搭子招募）

互助板块采用轻量信息发布模型，适用于失物招领和二手交易。两类内容共享“发布 → 审核 → 展示 → 私聊联系 → 发布者标记结束”的基础流程。平台不为互助板块维护订单、报名、公开回答或平台支付闭环；二手交易由买卖双方线下自行完成，平台不介入资金流转。

#### 互助发布内容状态说明

> 来源：[AI生成+人工修改]（说明：AI 整理互助发布内容状态表，人工确认审核可见性、驳回重提、撤回、已结束内容不可举报/删除/修改等规则）

| 状态 | 含义 | 适用场景 |
|------|------|----------|
| 审核中 | 用户提交内容后，等待审核 | 新发布的失物招领、二手商品 |
| 已发布 | 审核通过并公开展示 | 需求大厅可见内容 |
| 已结束 | 发布者标记已解决或已出；不再公开展示，不支持举报或删除 | 失物招领、二手交易 |
| 已驳回 | 审核未通过，不公开展示 | 审核失败内容 |
| 已删除 | 用户撤回已发布内容，或管理员删除举报成立的已发布违规内容，不再展示 | 用户撤回或管理员处理已发布违规内容 |

互助发布内容在审核中或已发布状态下，发布者可主动撤回；撤回后状态变为已删除，不再进入需求大厅展示。
审核中的内容仅发布者本人和管理员可见，其他用户不可见。审核驳回后，发布者可修改内容并重新提交审核。
已发布的互助内容不支持直接修改；如内容有误，发布者可撤回后重新发布。已发布内容可被其他用户举报，管理员审核举报成立后可删除。已结束内容不再公开展示，不支持举报、删除或修改。

| 服务板块 | 提供明确履约流程的校园代取服务，MVP 阶段仅支持代取，不支持代购或代办 |
| 代取服务 | 代取快递、外卖、已明确物品等，不要求接单方垫付本金 |
| 互助板块 | 提供轻量互助信息发布与私聊联系能力，MVP 阶段包括失物招领和二手交易 |
| 好评率 | 基于用户收到的代取服务评价计算的可信度指标 |
| CAS | Central Authentication Service，中央认证服务 |

### 6.2 支付系统说明（支付宝沙箱）

| 场景 | 支付方式 | 说明 |
|------|---------|------|
| 发布有报酬代取服务 | 发布方通过支付宝沙箱支付预付款 | 服务先进入待支付状态；发布者可继续支付或取消，超时未支付自动取消；支付成功后预付款暂存平台沙箱账户，服务进入审核中 |
| 确认完成有报酬代取服务 | 系统通过支付宝沙箱转账至接单方 | 发布方确认后触发转账 |
| 取消有报酬代取服务 | 系统通过支付宝沙箱退还预付款至发布方 | 原路退回 |
| 无报酬代取服务 | 不涉及资金流转 | 不填写报酬金额 |
| 二手交易 | 不接入平台支付 | 买卖双方线下自行交易 |

> **说明：** MVP阶段使用支付宝沙箱环境，不涉及真实资金。

### 6.3 MVP 范围说明

**本次迭代实现（Must + Should）：**

- 用户注册与学号认证（提交校园卡/学生证照片，管理员人工审核）
- 学号唯一性约束（每个学号只能绑定一个账号）
- 服务板块：代取服务完整流程（支持有报酬和无报酬服务，有报酬服务通过支付宝沙箱支付；不支持代购/代办）
- 互助板块：失物招领发布、浏览、私聊联系、标记已解决
- 互助板块：二手交易发布、浏览、私聊联系、标记已出，平台不介入资金流转
- 校区筛选（手动选择）
- 双方互评与好评率展示
- 站内即时通讯
- 管理员基础功能（封禁、内容审核、申诉处理）

**Could（MVP 阶段暂不实现，后续可选）：**

- 咨询问答、公开回答与最佳答案采纳

**本次迭代不实现（Won't）：**

- 真实第三方支付生产环境对接（微信支付/支付宝生产环境，MVP 不发生真实资金流转）
- 代购、代办及需要接单方垫付本金的服务
- 找搭子发布、报名与活动小组确认
- 学校官方认证 API 对接（MVP 采用管理员人工审核）
- 数据分析看板
- 兴趣匹配聊天（社交延伸，非核心）
- 多校区自动路由（先支持手动选择）
- 众包纠纷仲裁（“判官”系统）
- 进行中服务超时自动处理（具体阈值无用户共识）
- 高并发支持（MVP 支持 200 并发，后续按需扩容）

### 6.4 后续规划（本次不实现）

以下功能不在 MVP 范围内，计划在后续版本中根据用户反馈和资源情况逐步实现：

| 序号 | 功能 | 说明 | 触发条件/优先级 |
|------|------|------|----------------|
| 1 | **对接官方认证 API** | 接入学校统一身份认证系统（如 CAS），实现自动化学号实名认证，替代 MVP 阶段的管理员人工审核方式。 | 学校提供接口后优先接入 |
| 2 | **对接真实支付宝支付** | 将支付宝沙箱切换为真实支付宝支付，支持真实资金流转。MVP阶段使用沙箱环境验证支付流程。 | 沙箱验证通过后评估 |
| 3 | **代购/代办服务** | 支持本金预付、实际费用凭证、多退少补和失败退款流程。 | 支付与纠纷规则成熟后评估 |
