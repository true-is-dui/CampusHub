## 1. 引言

### 1.1 目的

本文档旨在定义“校园互助服务平台”的完整功能性和非功能性需求。该平台旨在连接校园内的学生，使他们能够发布和响应各类互助需求，从而构建一个安全、高效、可信的校园服务生态系统。本文档是项目团队进行设计、实现和测试的主要依据。

### 1.2 文档约定

- **优先级定义（MoSCoW）：**
    - `[Must]`：必须实现，否则产品不可用。
    - `[Should]`：应该实现，除非有明确理由。
    - `[Could]`：可以实现，视资源与时间而定。
- **需求编号规则：** `FR-[模块]-[序号]`（功能需求）、`NFR-[类别]-[序号]`（非功能需求）。
- **标注规则：** 每条需求末尾标注 `[AI生成]` 或 `[人工补充]`。

### 1.3 预期的读者和阅读建议

- **项目经理/产品经理：** 关注第 2 章（总体描述）和第 3 章（功能需求）。
- **开发人员/测试人员：** 重点关注第 3 章（功能需求）和第 4 章（非功能需求）。
- **UI/UX 设计师：** 参考第 3.1 条用户界面需求及第 3.2 条各功能模块逻辑。

### 1.4 产品的范围

该产品是一个基于 Web 或移动端的校园互助平台。核心范围包括：用户注册与认证、多类型互助需求的发布与接单、站内即时通讯、信用评价体系、交易担保与支付（可选）、后台管理功能。产品初期聚焦于快递代取、学习辅导、二手交易和组队匹配四大核心场景。

### 1.5 参考文献

- IEEE Std 830-1998: IEEE Recommended Practice for Software Requirements Specifications.

## 2. 总体描述

### 2.1 产品视角

该产品是一个独立运行的软件系统，不依赖于任何现有的大型系统，但需与第三方支付网关（如微信支付、支付宝）和短信服务商（SMS API）对接。用户通过个人移动设备或 PC 浏览器访问。

### 2.2 产品功能（高层概要）

1.  **用户管理：** 注册、登录、个人资料管理、学号实名认证。
2.  **需求大厅：** 发布互助任务，按分类、距离、时间筛选浏览任务。
3.  **任务流：** 抢单/接单机制，任务状态流转（发布-进行中-已完成-已取消）。
4.  **支付与结算：** 支持线上支付（托管模式）与线下现金交易，平台货币（积分）体系。
5.  **信用体系：** 双方互评，信誉分计算，历史记录展示。
6.  **即时通讯：** 订单内的私聊沟通。
7.  **后台管理：** 用户封禁、内容审核、纠纷仲裁、数据统计。

### 2.3 用户特征

| 用户角色 | 特征描述 | 使用频率 | 技术水平 |
| :--- | :--- | :--- | :--- |
| **学生（需求方）** | 时间紧张、有特定需求的在校大学生。 | 高（每周 1-3 次） | 中 |
| **学生（接单方）** | 愿意利用碎片时间赚取酬劳或积分的勤工俭学学生。 | 高（每周多次） | 中 |
| **管理员** | 学校相关组织或运营方人员，负责平台秩序。 | 中 | 高 |

### 2.4 运行环境

- **客户端：** iOS 11+/Android 8+ 微信小程序或 H5 页面，主流浏览器（Chrome, Safari）。
- **服务器端：** Linux 环境，支持 Node.js/Python/Java 运行环境，MySQL/PostgreSQL 数据库，Redis 缓存。

### 2.5 设计和实现上的限制

- **认证限制：** 必须使用官方教务系统 API 或人工审核学生证来完成学号实名认证。
- **支付牌照：** 若涉及资金池托管，需严格遵守国家金融监管规定，或接入有牌照的第三方支付平台。
- **隐私合规：** 必须符合《个人信息保护法》，用户手机号、位置需脱敏处理。

### 2.6 假设和依赖

- **假设：** 用户拥有智能手机并熟悉基本操作；校园内有稳定的网络覆盖。
- **依赖：** 依赖学校提供学生身份验证接口；依赖第三方地图服务（如高德/百度）提供位置服务。

## 第二步：人工调研发现的需求

> 说明：本节记录的是问卷和访谈中出现的候选需求，不代表最终全部采纳。最终需求需结合 10 周 MVP 周期、团队实现能力、合规风险和后续评审结果进行收敛。

### 人工调研的核心需求摘要

| 模块 | 人工调研发现的需求 |
|------|------------------|
| 核心场景 | 跑腿代取（~80%）、咨询问答（~70%）、二手交易（~60%）、失物招领（~40%）、找搭子/组队（~30%）、学习辅导/资料分享（~25%） |

| 功能细节 | 校区筛选（多校区）、兴趣标签匹配（找搭子）、匿名发帖/树洞、任务状态推送通知、失物招领（无报酬）、咨询问答的“专家认证” |

## 第三步：对比分析

| 对比维度 | AI 生成的需求 | 人工调研发现的需求 | 分析 |
|---------|-------------|----------------|------|
| AI 正确覆盖的需求 | 学号实名认证、任务发布与接单、任务状态流转、双方互评、信誉分、站内沟通、后台管理 | — | AI 准确识别了平台的核心骨架：实名认证建立信任、任务闭环流程、信用评价体系。这些在问卷和访谈中均被高频验证。 |
| AI 遗漏的需求 | — | 失物招领、找搭子兴趣标签、咨询问答专家认证、匿名树洞、校区筛选、信用承诺金、双维度信誉分（交易/回答）、实时推送通知 | **为什么 AI 会遗漏？**<br><br>1. 场景泛化：AI 将“互助”简化为“任务-报酬”模型，遗漏失物招领（无报酬）、找搭子（社交属性）等非交易场景。<br><br>2. 校园细节缺失：多校区筛选、兴趣标签是校园特有需求。<br><br>3. 匿名需求被忽视：AI 侧重实名，但部分用户需要匿名。<br><br>4. 信任机制粗糙：未区分“交易诚信”与“回答质量”。 |
| AI 捏造的需求 | 5%手续费、1:10积分兑换、24h/2h超时阈值、60分信誉阈值、支付网关对接 | — | **这些需求真的存在吗？**<br><br>1. 部分存在但细节不实：用户接受积分，但具体比例/费率无共识。<br><br>2. 部分超出范围：支付托管有提及，但属高级功能（Could），AI 标为 Must/Should。<br><br>3. 具体阈值无依据：时间/分数阈值为 AI 随意猜测。 |
| AI 表述有歧义的需求 | “我的钱包”、“信誉分低于60分”、“2秒加载”、“500并发”、“接单” | — | **歧义在哪里？**<br><br>1. “钱包”：积分还是现金？能否提现？<br><br>2. “60分”：初始值？加减规则？依据？<br><br>3. “2秒”：首屏还是完整渲染？网络条件？<br><br>4. “500并发”：在线用户还是请求数？<br><br>5. “接单”：抢单还是需求方选择？ |

## 第四步：后续评审收敛结果

在项目进入 P4 后，团队对 P1 需求文档进行了回溯评审。以下内容是对前述 AI 初稿和人工调研候选需求的进一步收敛，不改变第一步中 AI 原始输出的记录性质。

| 收敛项 | 处理结果 | 原因 |
|------|---------|------|
| 产品形态 | 收敛为电脑端浏览器 Web 平台 | 团队实际计划开发电脑端网页，原移动端/小程序表述与实现方向不一致 |
| 积分体系 | 废除积分、签到积分、认证送积分、积分提现等设计 | 积分兑换、提现比例和激励规则缺少调研共识，且会增加实现复杂度 |
| 支付方案 | 仅跑腿有报酬任务使用支付宝沙箱验证预付款、转账和退款流程；MVP 不发生真实资金流转 | 保留跑腿报酬保障流程，同时规避真实支付生产环境和合规风险 |
| 二手交易 | 线下自行交易，平台不提供担保支付或退款 | 二手交易资金监管复杂，MVP 阶段不介入更可控 |
| 实名认证 | 用户参与平台活动前需提交学号、姓名和校园卡/学生证照片，由管理员人工审核 | 兼顾身份可信和 MVP 可实现性，不依赖学校统一认证接口 |
| 内容审核 | 仅保留管理员人工审核，发布内容审核通过后才公开 | 自动审核实现成本和准确性风险较高，人工审核更适合 MVP |
| 需求大厅 | 按板块浏览，不做跨板块综合大厅；支持当前板块内关键词搜索、校区筛选和发布时间倒序 | 降低搜索和推荐复杂度，同时保留基本浏览效率 |
| 评价体系 | 废除信誉分和信用扣分，保留跑腿评价与好评率 | 信誉分计算规则复杂且缺少依据，好评率更直观可解释 |
| 私聊功能 | 跑腿、失物招领、找搭子、二手交易支持私聊；咨询问答不提供私聊 | 匹配各业务的实际联系需求，避免问答场景转入私聊后破坏公开问答价值 |
| 举报与删除 | 用户只能举报自己可见内容；已结束主业务内容不可举报或删除；评价可举报并由管理员删除违规评价 | 简化内容治理边界，避免对已结束业务反复修改影响状态稳定性 |

## 四、结论与启示

| 维度 | 结论 |
|------|------|
| AI 的优势 | 快速生成结构规范、逻辑完整的 SRS 初稿，覆盖通用功能骨架，可节省约 40-50% 基础撰写时间 |
