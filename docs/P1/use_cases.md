## 二、核心用例详细描述

### 2.1 用例 UC-03：发布跑腿任务

| 项目 | 内容 |
|------|------|
| **用例名称** | 发布跑腿任务 |
| **参与者** | 普通用户（已完成学号认证） |
| **前置条件** | 1. 用户已登录系统<br>2. 用户已完成学号实名认证 |
| **后置条件** | 1. 任务提交成功，等待管理员内容审核<br>2. 有报酬任务的预付款暂存平台中间账户（支付宝沙箱）<br>3. 任务状态为”审核中” |

**基本流：**

| 步骤 | 操作 |
|------|------|
| 1 | 用户进入“发布任务”页面 |
| 2 | 用户选择任务类型为“跑腿代取” |
| 3 | 用户填写任务信息：取件地点、送达地点、物品描述、任务报酬类型、接单截止时间、校区；若选择有报酬任务，需填写报酬金额（1-200 元） |
| 4 | 用户确认发布 |
| 5 | 若为有报酬任务，系统引导用户通过支付宝沙箱完成预付款 |
| 6 | 有报酬任务支付成功后，系统生成任务，状态设为”审核中” |
| 7 | 若为无报酬任务，系统直接生成任务，状态设为”审核中” |
| 8 | 系统向用户返回提交成功提示 |
| 9 | 管理员审核通过后，任务状态变为”待接单”并出现在需求大厅列表 |

**替代流：**

| 步骤 | 分支 | 处理 |
|------|------|------|
| 3a | 用户填写信息不完整 | 系统提示”请填写完整信息”，不提交 |
| 5a | 有报酬任务支付失败或用户取消支付 | 系统提示”支付未完成，任务暂未发布”，不生成任务 |
| 6a | 支付回调异常 | 系统提示”发布失败，请重试”，已支付金额原路退回 |
| 9a | 管理员审核驳回 | 系统通知发布方，任务不进入需求大厅；有报酬任务的预付款原路退回 |
| 3b | 用户点击取消 | 返回上一页，不保存信息 |

### 2.2 用例 UC-04：接单并完成跑腿任务

| 项目 | 内容 |
|------|------|
| **用例名称** | 接单并完成跑腿任务 |
| **参与者** | 普通用户（发布方、接单方，均已完成学号认证） |
| **前置条件** | 1. 接单方已登录系统<br>2. 接单方已完成学号实名认证<br>3. 存在“待接单”状态的任务<br>4. 接单方不是该任务的发布者 |
| **后置条件** | 1. 接单成功后任务状态变更为“进行中”，其他用户不可再接此单<br>2. 接单方上传完成凭证后，发布方可确认完成<br>3. 发布方确认完成后任务状态变更为“已完成”<br>4. 有报酬任务通过支付宝沙箱将预付款转账至接单方；无报酬任务不涉及资金转移 |

**基本流：**

| 步骤 | 操作 |
|------|------|
| 1 | 用户在需求大厅浏览任务列表 |
| 2 | 用户点击某个任务查看详情 |
| 3 | 用户点击“接单”按钮 |
| 4 | 系统检查任务状态是否为“待接单” |
| 5 | 系统将任务状态更新为“进行中” |
| 6 | 系统记录接单方信息 |
| 7 | 系统开放任务双方的站内私聊入口 |
| 8 | 系统向发布方推送“已有人接单”通知 |
| 9 | 系统向接单方返回“接单成功”提示 |
| 10 | 接单方完成跑腿任务后上传完成凭证照片 |
| 11 | 系统通知发布方查看完成凭证并确认任务 |
| 12 | 发布方确认任务完成 |
| 13 | 系统将任务状态更新为“已完成” |
| 14 | 若任务有报酬，系统通过支付宝沙箱将预付款转账至接单方；若任务无报酬，系统不进行资金处理 |
| 15 | 系统开放双方互评入口 |

**替代流：**

| 步骤 | 分支 | 处理 |
|------|------|------|
| 4a | 任务已被其他人接走 | 系统提示“任务已被接走”，返回任务列表 |
| 4b | 任务已超过接单截止时间 | 系统提示“任务已超过接单截止时间”，返回任务列表 |
| 5a | 状态更新失败 | 系统提示“接单失败，请重试” |
| 3a | 用户是任务发布者 | 系统提示“不能接自己的任务”，禁止接单 |
| 10a | 未上传完成凭证 | 系统提示“请上传完成凭证”，不可提交完成 |
| 10b | 完成凭证不符合要求 | 系统提示图片数量、大小或格式错误，要求重新上传 |
| 12a | 发布方暂不确认完成 | 任务保持“进行中”状态，双方可继续沟通；如有争议，可提交任务异常申诉 |
| 12b | 发布方长时间未确认完成 | 接单方可提交任务异常申诉，由管理员处理 |

### 2.3 用例 UC-14：互评

| 项目 | 内容 |
|------|------|
| **用例名称** | 互评 |
| **参与者** | 普通用户（已完成任务的双方） |
| **前置条件** | 1. 任务状态为“已完成”<br>2. 评价方尚未对该任务进行评价 |
| **后置条件** | 1. 评价记录保存到数据库<br>2. 被评价方的好评率重新计算<br>3. 任务双方的评价记录公开可见 |

**基本流：**

| 步骤 | 操作 |
|------|------|
| 1 | 用户进入“我的任务”页面 |
| 2 | 用户点击“待评价”标签 |
| 3 | 用户选择某个已完成的任务 |
| 4 | 用户选择评价等级（好评/中评/差评） |
| 5 | 用户填写文字评价（好评和中评选填，差评必填理由） |
| 6 | 用户提交评价 |
| 7 | 系统保存评价记录 |
| 8 | 系统更新被评价方的好评率 |
| 9 | 系统向被评价方推送“收到新评价”通知 |

**替代流：**

| 步骤 | 分支 | 处理 |
|------|------|------|
| 4a | 用户选择“差评”但未填写理由 | 系统提示“差评需填写理由”，不提交 |
| 5a | 用户提交后 | 用户可撤回自己发出的评价；如对收到的评价有异议，需通过申诉流程处理 |
| 2a | 用户未评价 | 系统不生成评价记录，该用户的未评价行为不计入对方好评率 |
| 6a | 评价提交失败 | 系统提示“评价失败，请重试” |
| 4b | 用户点击取消 | 返回上一页，不保存评价 |
