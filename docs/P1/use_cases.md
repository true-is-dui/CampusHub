## 二、核心用例详细描述

### 2.1 用例 UC-03：发布代取服务

| 项目 | 内容 |
|------|------|
| **用例名称** | 发布代取服务 |
| **参与者** | 普通用户（已完成学号认证） |
| **前置条件** | 1. 用户已登录系统<br>2. 用户已完成学号实名认证 |
| **后置条件** | 1. 无报酬服务提交成功后，状态为“审核中”，等待管理员内容审核<br>2. 有报酬服务提交成功后，先生成发布者可见的“待支付”服务和支付宝沙箱预付款入口<br>3. 有报酬服务支付成功后，预付款暂存平台中间账户（支付宝沙箱），服务状态变为“审核中” |

**基本流：**

| 步骤 | 操作 |
|------|------|
| 1 | 用户进入“发布服务”页面 |
| 2 | 用户选择服务类型为“代取” |
| 3 | 用户填写服务信息：取件地点、送达地点、物品描述、取件凭证、报酬类型、接单截止时间、校区；若选择有报酬服务，需填写报酬金额（1-200 元） |
| 4 | 用户确认发布 |
| 5 | 若为有报酬服务，系统生成“待支付”代取服务，创建支付宝沙箱预付款入口，并引导用户完成预付款 |
| 6 | 待支付服务仅发布者本人可见，不进入服务大厅和管理员内容审核队列 |
| 7 | 有报酬服务支付成功后，系统将服务状态设为“审核中”，并创建内容审核记录 |
| 8 | 若为无报酬服务，系统直接生成服务，状态设为“审核中”，并创建内容审核记录 |
| 9 | 系统向用户返回提交成功提示 |
| 10 | 管理员审核通过后，服务状态变为“待接单”并出现在需求大厅服务板块 |

**替代流：**

| 步骤 | 分支 | 处理 |
|------|------|------|
| 3a | 用户填写信息不完整 | 系统提示”请填写完整信息”，不提交 |
| 5a | 有报酬服务支付失败 | 服务保持“待支付”状态，发布者可继续支付或手动取消 |
| 5b | 发布者在待支付状态下取消支付 | 系统将服务状态改为“已取消”，服务不进入审核和需求大厅 |
| 5c | 待支付服务超过支付有效期仍未支付 | 系统自动将服务状态改为“已取消”，服务不进入审核和需求大厅 |
| 7a | 支付回调异常 | 系统保留服务为“待支付”并提示稍后查询支付结果；若确认已扣款但无法入账，由支付记录处理退款或人工核查 |
| 10a | 管理员审核驳回 | 系统通知发布方，服务不进入需求大厅；有报酬服务的预付款原路退回 |
| 3b | 用户点击取消 | 返回上一页，不保存信息 |

### 2.2 用例 UC-04：接单并完成代取服务

| 项目 | 内容 |
|------|------|
| **用例名称** | 接单并完成代取服务 |
| **参与者** | 普通用户（发布方、接单方，均已完成学号认证） |
| **前置条件** | 1. 接单方已登录系统<br>2. 接单方已完成学号实名认证<br>3. 存在“待接单”状态的代取服务<br>4. 接单方不是该服务的发布者 |
| **后置条件** | 1. 接单成功后服务状态变更为“进行中”，其他用户不可再接此单<br>2. 接单方上传完成凭证后，发布方可确认完成<br>3. 发布方确认完成后服务状态变更为“已完成”<br>4. 有报酬服务通过支付宝沙箱将预付款转账至接单方；无报酬服务不涉及资金转移 |

**基本流：**

| 步骤 | 操作 |
|------|------|
| 1 | 用户在需求大厅服务板块浏览代取服务列表 |
| 2 | 用户点击某个代取服务查看详情 |
| 3 | 用户点击“接单”按钮 |
| 4 | 系统检查服务状态是否为“待接单” |
| 5 | 系统将服务状态更新为“进行中” |
| 6 | 系统记录接单方信息 |
| 7 | 系统向接单方展示取件凭证，并开放服务双方的站内私聊入口 |
| 8 | 系统向发布方推送“已有人接单”通知 |
| 9 | 系统向接单方返回“接单成功”提示 |
| 10 | 接单方完成代取服务后上传完成凭证照片 |
| 11 | 系统通知发布方查看完成凭证并确认服务 |
| 12 | 发布方确认服务完成 |
| 13 | 系统将服务状态更新为“已完成” |
| 14 | 若服务有报酬，系统通过支付宝沙箱将预付款转账至接单方；若服务无报酬，系统不进行资金处理 |
| 15 | 系统开放双方互评入口 |

**替代流：**

| 步骤 | 分支 | 处理 |
|------|------|------|
| 4a | 服务已被其他人接走 | 系统提示“服务已被接走”，返回服务列表 |
| 4b | 服务已超过接单截止时间 | 系统提示“服务已超过接单截止时间”，返回服务列表 |
| 5a | 状态更新失败 | 系统提示“接单失败，请重试” |
| 3a | 用户是服务发布者 | 系统提示“不能接自己的服务”，禁止接单 |
| 10a | 未上传完成凭证 | 系统提示“请上传完成凭证”，不可提交完成 |
| 10b | 完成凭证不符合要求 | 系统提示图片数量、大小或格式错误，要求重新上传 |
| 12a | 发布方暂不确认完成 | 服务保持“进行中”状态，双方可继续沟通；如有争议，可提交代取服务异常申诉 |
| 12b | 发布方长时间未确认完成 | 接单方可提交代取服务异常申诉，由管理员处理 |

### 2.3 用例 UC-14：互评

| 项目 | 内容 |
|------|------|
| **用例名称** | 互评 |
| **参与者** | 普通用户（已完成代取服务的双方） |
| **前置条件** | 1. 服务状态为“已完成”<br>2. 评价方尚未对该服务进行评价 |
| **后置条件** | 1. 评价记录保存到数据库<br>2. 被评价方的好评率重新计算<br>3. 服务双方的评价记录公开可见 |

**基本流：**

| 步骤 | 操作 |
|------|------|
| 1 | 用户进入“我的服务”页面 |
| 2 | 用户点击“待评价”标签 |
| 3 | 用户选择某个已完成的代取服务 |
| 4 | 用户选择评价等级（好评/中评/差评） |
| 5 | 用户填写文字评价（好评和中评选填，差评必填理由） |
| 6 | 用户提交评价 |
| 7 | 系统保存评价记录 |
| 8 | 系统更新被评价方的好评率 |
| 9 | 系统向被评价方推送“收到新评价”通知 |

**替代流：**

| 步骤 | 分支 | 处理 |
|------|------|------|
| 4a | 用户选择“差评”但未填写理由 | 系统提示“差评需填写理由”，不提交 |
| 5a | 用户提交后 | 用户可撤回自己发出的评价；如对收到的评价有异议，需通过申诉流程处理 |
| 2a | 用户未评价 | 系统不生成评价记录，该用户的未评价行为不计入对方好评率 |
| 6a | 评价提交失败 | 系统提示“评价失败，请重试” |
| 4b | 用户点击取消 | 返回上一页，不保存评价 |
