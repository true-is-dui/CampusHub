## Prompt #1：直接提问（成员A：姚圳锴）

**工具：** DeepSeek
**日期：** 2026-04-28

**Prompt：**
```
我要做一个校园互助平台，包含跑腿代取、失物招领、找搭子、二手交易、咨询问答等功能，用户端是H5移动网页，推荐什么架构？
```

**AI回答摘要：**
推荐微服务架构。将每个业务模块拆分为独立微服务（用户服务、任务服务、失物招领服务、搭子服务、二手交易服务、问答服务），使用Spring Cloud作为微服务框架，Nacos做服务注册与发现，Gateway做API网关，Docker + K8s部署。前端建议使用Vue 3构建H5移动网页。

**人工评估：** 质量2/5。未询问团队规模和周期，直接推荐最复杂方案。对4人10周项目完全不适用。

## Prompt #2：提供约束后提问（成员B：张皓）

**工具：** DeepSeek
**日期：** 2026-04-28

**Prompt：**
```
我需要为一个校园互助平台做技术选型。以下是约束条件：
- 团队4人（都是大二/大三学生）
- 开发周期10周
- MVP功能：用户认证、跑腿代取、失物招领、找搭子、二手交易、咨询问答、即时通讯、信用评价、后台管理
- 目标用户为校内学生，初期并发量约200人
- 不涉及支付和资金流转
- 用户端为H5移动网页，管理端为Web后台

请推荐合适的架构方案，并说明理由。
```

**AI回答摘要：**
推荐"前后端分离 + 后端单体分层架构"。理由：(1) 4人团队不适合维护多服务；(2) 10周内单体开发效率最高；(3) 200并发无需微服务的弹性伸缩。技术栈建议：Spring Boot + MySQL + Redis + WebSocket。分层方式：Controller → Service → Repository。同时建议做好模块间接口定义，未来可按需拆分为微服务。

**人工评估：** 质量4.5/5。约束充分后，AI的建议与团队最终决策高度一致。WebSocket建议有价值。

## Prompt #3：要求对比分析（成员C：谢易轩）

**工具：** DeepSeek
**日期：** 2026-04-29

**Prompt：**
```
校园互助平台（用户认证、任务发布接单、即时通讯、评价系统），请对比单体分层架构和微服务架构的优劣，从开发成本、部署复杂度、扩展性、团队要求、运维成本五个维度分析，并给出推荐。
```

**AI回答摘要：**

| 维度 | 单体分层 | 微服务 |
|------|---------|--------|
| 开发成本 | 低 | 高 |
| 部署复杂度 | 简单 | 复杂（需容器编排） |
| 扩展性 | 有限（整体扩容） | 高（独立扩容） |
| 团队要求 | 低（全栈开发） | 高（需DevOps） |
| 运维成本 | 低 | 高（多服务监控） |

结论：团队≤5人选单体；团队≥10人选微服务。

**人工评估：** 质量4/5。对比结构完整，条件性结论合理。缺少"从单体到微服务的演进路径"维度。

## Prompt #4：角色扮演（成员D：毛嘉和）

**工具：** DeepSeek
**日期：** 2026-04-29

**Prompt：**
```
你现在是一名有10年经验的软件架构师。请为一个校园互助平台设计架构方案。平台功能包括跑腿代取、失物招领、找搭子、二手交易、咨询问答。请从架构风格、技术栈、模块划分、质量属性四个方面给出建议。
```

**AI回答摘要：**
建议"分层架构 + 事件驱动补充"。核心业务流程用分层，通知/支付状态变更用事件驱动。技术栈：Spring Boot + MySQL + Redis + RabbitMQ + WebSocket + Elasticsearch。模块分为6个核心域。质量属性建议包括可测试性（单元测试覆盖≥80%）、可维护性（统一代码规范）、安全性（JWT + HTTPS + 数据脱敏）。

**人工评估：** 质量3.5/5。引入了过多中间件（RabbitMQ、Elasticsearch），对MVP过度设计。但质量属性分析有价值。角色扮演让AI追求"专业完整性"而忽视了"可行性"。

## Prompt #5：ADR生成

**工具：** DeepSeek
**日期：** 2026-04-30

**Prompt：**
```
校园互助平台（4人团队，10周MVP），我们决定采用：
1. 前后端分离+后端单体分层架构
2. H5移动网页作为用户端
3. MySQL + Redis作为数据存储
4. 自建WebSocket实现IM

请为以上4条决策编写ADR，每条含：状态、背景、决策、理由、后果。
```

**AI回答摘要：**
AI生成了4条ADR的结构，但：(1) "后果"部分只有正面影响，缺少负面和风险；(2) 理由偏通用模板化（如"Spring Boot生态成熟"），缺少针对本项目的具体论证；(3) 没有与其他方案的对比分析。

**人工评估：** 质量3.5/5。框架可用，但内容需要大量补充。ADR的"权衡"本质没有被充分体现——AI把ADR写成了"决策宣传"而非"决策记录"。

**人工修订要点：**
- 补充每条决策的负面影响和风险
- 增加AI辅助记录部分（AI初稿 vs 人工修订对比）
- 增加具体数值和场景使决策可验证
- 补充"不选其他方案的原因"

## Prompt #6：技术选型建议

**工具：** DeepSeek
**日期：** 2026-04-30

**Prompt：**
```
校园互助平台技术选型建议：
- 用户端：H5移动网页
- 管理端：Web后台
- 后端框架：Java技术栈
- 功能：用户管理、任务发布接单、即时通讯、评价系统、后台管理
- 约束：4人团队、10周、200并发、无支付

请推荐完整技术栈（前端框架、后端框架、数据库、缓存、消息队列、部署方式等），每项说明选择理由。
```

**AI回答摘要：**
推荐技术栈：Vue 3 + Spring Boot + MySQL + Redis + Docker。建议不使用消息队列。部署用Docker Compose单机部署。API文档用Swagger。

**人工评估：** 质量4/5。推荐整体合理。但AI没有说明"为什么不选其他方案"（如为什么不用uni-app跨平台、为什么不用MongoDB），这部分由人工补充。

## P4回溯补充：P2架构文档修订中的AI协作记录

> 说明：本节为P4回溯评审阶段对P2文档的补充记录，不覆盖P2当时的Prompt与回答摘要。

| 讨论点 | AI建议或提醒 | 人工纠正或确认 | 文档修订结果 |
|--------|--------------|----------------|--------------|
| P1 v2.1 范围同步 | 提醒 P2 仍把失物招领、二手交易、IM、举报/申诉写成 MVP 模块 | 人工确认这些均为 Could，不进入核心架构 | 模块划分收缩为用户、实名审核、代取服务、支付沙箱、文件、评价、通知 |
| 前端产品形态与技术栈 | 建议继续采用电脑端 Web 单页应用，并使用 Vue 3、Vite、Element Plus、Vue Router、Pinia、Axios | 人工确认使用 JavaScript，不引入 TypeScript；学生端和管理端共用一个 SPA | 架构文档和 ADR-002 保持电脑端 Web SPA 方案 |
| 后端技术栈收敛 | 提醒 Redis、WebSocket、MQ、MongoDB、Elasticsearch 都需要重新评估是否支撑 Must 需求 | 人工确认 Spring Boot、MyBatis Plus、MySQL、本地文件存储和支付宝沙箱为必选；Redis 仅可选 | 技术选型表和 ADR-003 完成二次收敛 |
| 代取状态流转 | 发现旧 P2 中“支付成功后进入审核中”与 P1 v2.1 冲突 | 人工确认 MVP 不做普通内容审核，支付成功后进入待接单 | 架构文档新增关键状态规则，删除内容审核链路 |
| 支付模块边界 | AI提醒支付模块不应维护完整代取业务对象 | 人工明确代取服务模块维护 `pickupId` 与 `paymentId` 关系，支付模块只维护预付款、退款、结算记录 | 模块接口保留 `createPrepay`、`queryPaymentStatus`、`refundPayment`、`settlePayment` |
| 文件模块边界 | AI提醒取件凭证和完成凭证需要权限控制 | 人工确认文件模块不直接面向前端开放，业务模块鉴权后调用文件读取 | 文件模块职责保留头像、认证材料、取件凭证和完成凭证 |
| 管理模块收缩 | AI指出旧管理模块包含内容审核、封禁、举报、申诉等宽泛职责 | 人工确认当前管理模块只负责实名认证审核 | ADR-001 和架构模块表同步收缩管理职责 |
| 通知方案调整 | AI指出 WebSocket IM 决策已不匹配 P1 v2.1 | 人工确认站内通知采用数据库通知记录，不引入 WebSocket/MQ | ADR-004 改为“数据库通知记录 + 前端查询/轻量轮询” |
| 非功能需求闭环 | AI提醒 P2 架构文档应只映射 P1 当前非功能需求 | 人工确认不再覆盖聊天、举报、跨板块大厅等旧能力 | 质量属性章节改为代取大厅性能、安全、日志和桌面端兼容 |
| 架构图表达 | AI建议继续使用 Mermaid，便于回溯调整 | 人工判断 Mermaid 比静态图片更易维护 | 架构概览图和部署架构图均保留 Mermaid 表达 |
