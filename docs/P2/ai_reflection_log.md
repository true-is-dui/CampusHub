## 2. AI 助力分析（Q1）

**本阶段 AI 最大的贡献是：**

快速生成候选方案的对比框架和ADR模板结构。当提供了充分的约束条件（团队规模、开发周期、功能范围）后，AI给出的架构建议与团队最终决策高度一致，节省了约3-4小时的调研和文档搭建时间。

**不同 Prompt 策略下 AI 回答质量的差异：**

| Prompt策略 | AI表现 | 关键差异 |
|-----------|--------|---------|
| 直接提问（无约束） | 推荐微服务，不切实际 | AI默认假设"充足资源+无限时间" |
| 提供详细约束 | 推荐单体分层，务实可行 | 约束条件改变了AI的优化目标 |
| 要求对比分析 | 给出条件性结论，结构完整 | AI表现出较好的对比推理能力 |
| 角色扮演 | 更关注完整性，引入过度设计 | 角色设定让AI追求"专业感"而非"可行性" |

**核心发现：** AI的输出质量不是由"Prompt技巧"决定的，而是由"约束信息的充分程度"决定的。提供清晰的项目约束（团队规模、时间、功能范围）比精心设计Prompt话术更有效。

## 3. AI 误导分析（Q2）

**AI 在架构建议中的典型问题：**

- **问题1：默认推荐"技术最先进"方案而非"当前最合适"方案**
  - 如何发现：无约束条件下AI一致推荐微服务；加上4人10周约束后立即改为单体
  - 如何修正：先明确约束条件再提问；将AI输出作为"理想方案"参考，由人工根据现实约束做减法

- **问题2：倾向于引入过多中间件（"重型架构"偏好）**
  - 如何发现：AI在角色扮演模式下推荐引入RabbitMQ（消息队列）+ Elasticsearch（全文搜索）+ MongoDB（文档存储），"全家桶"式推荐
  - 如何修正：逐条评估每个中间件的必要性——RabbitMQ用Spring Event替代，Elasticsearch用MySQL索引替代，MongoDB暂不引入。每多一个中间件就多一套运维成本，MVP阶段"够用就好"

- **问题3：ADR初稿缺少"后果"维度的双向分析**
  - 如何发现：AI生成的ADR只有"正面影响"，没有"负面影响"和"风险"，读起来像推销文案而非决策记录
  - 如何修正：为每条ADR补充了负面影响和需要关注的风险，使决策记录真实反映权衡过程

- **问题4：AI的对比分析缺少"时间约束下的可行性"维度**
  - 如何发现：AI的架构对比表只有开发复杂度、可维护性、可扩展性、性能等通用维度，没有"10周内能否完成"
  - 如何修正：在所有对比表中新增"10周内可行性"作为独立维度，这是本项目最关键的评价标准

## 4. Prompt 改进计划（Q3）

**上阶段（P1）提出的改进措施，验证结果：**

| P1改进措施 | P2验证结果 | 效果 |
|-----------|-----------|------|
| 增加范围边界 | ✅ 有效。在Prompt中明确"MVP功能范围"和"H5用户端"后，AI不再推荐支付网关等非必需组件，前端建议也更具体 | 显著 |
| 增加开发约束 | ✅ 最有效。"4人10周"成为改变AI推荐方向的核心变量 | 非常显著 |
| 提供角色示例 | ⚠️ 部分有效。架构选型场景下角色示例不如约束条件影响大 | 一般 |
| 要求标注来源和优先级 | ✅ 有效。AI生成的ADR在要求标注后质量提升 | 中等 |

**本阶段新的改进措施：**

| 发现的问题 | 下阶段改进措施 |
|-----------|--------------|
| AI偏好"重型架构" | 在Prompt中明确要求"最小化技术栈，每引入一个新组件需说明必要性" |
| AI的ADR缺少风险分析 | 在Prompt中要求"每条决策必须包含正面影响、负面影响、风险三个维度" |
| 不同Prompt策略下答案差异大 | 下阶段固定使用"约束充分+要求对比"的Prompt模板，减少策略探索时间 |
| AI角色扮演容易过度设计 | 不再使用角色扮演策略；直接提供约束条件即可获得务实的方案 |

## 5. 本阶段的核心工程判断

**决策内容：**

在架构辩论赛中，团队否决了AI（无约束条件下）推荐的微服务方案，也否决了AI（角色扮演模式下）推荐的"分层+事件驱动+RabbitMQ"混合方案，选择最简洁的单体分层架构。

**为什么 AI 无法做这个决策：**

1. **AI不知道"够用就好"的工程哲学：** AI的优化目标似乎是"技术完备性"——数据库要选最合适的、消息队列要有、搜索引擎要有。但工程上"最简单的可行方案"往往是最优解。AI无法理解"多引入一个中间件，就多一个可能挂掉的东西"这种运维直觉。

2. **AI没有"技术债务"的概念：** AI推荐RabbitMQ、Elasticsearch时没有考虑——引入这些组件后，谁来维护？谁来在凌晨3点处理RabbitMQ宕机？4人团队没有专职运维，每多一个组件就多一份风险。

3. **AI缺乏"团队心理模型"：** AI不知道这4个人的技术水平、合作模式、学习速度。它推荐了一个"理论上最优"的方案，但不知道这个方案会压垮这个具体团队的工作量。这正是架构负责人需要做的判断——不是选"最好的架构"，而是选"这个团队能在10周内做完的架构"。

4. **AI的分层推荐是正确的，但"为什么对"AI说不清楚：** 当提供约束后，AI正确地推荐了单体分层。但AI的理由是通用模板（"适合中小团队"、"开发效率高"），缺少针对本项目的具体分析（如"需求大厅的缓存策略"、"WebSocket在单体中的连接管理"等）。这说明AI能给出正确方向，但无法提供深度工程论证。
