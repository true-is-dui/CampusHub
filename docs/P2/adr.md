## ADR-001：采用单体分层架构而非微服务架构

### 状态
已接受

### 背景
项目需要为校园互助平台选择整体架构风格。系统包含用户管理、跑腿代取、失物招领、找搭子、二手交易、咨询问答、即时通讯、信用评价、后台管理等9大业务模块。团队4人，开发周期10周，MVP阶段目标并发量200人。

### 决策
我们决定采用**前后端分离 + 后端单体分层架构**（Controller-Service-Repository），而非微服务架构。

### 理由
1. **团队规模与生产力：** 4人团队维护多个微服务会导致大量时间花在服务间通信、配置管理、本地调试上，而非业务开发
2. **开发周期约束：** 10周内需要完成MVP，微服务的基础设施（服务发现、配置中心、链路追踪、CI/CD多服务流水线）搭建至少需要2-3周
3. **并发量不构成拆分驱动力：** 200并发完全可以通过单实例+缓存解决，无需微服务的独立扩缩容能力
4. **模块化设计预留演进路径：** 分层架构内部通过良好的模块边界设计，允许未来将高频模块（如IM服务、任务服务）逐步拆分为独立服务
5. **团队技术储备：** 团队在课程中系统学习了Spring Boot分层架构，无微服务实战经验

### 后果
- **正面影响：** 开发效率高，本地调试简单，部署运维成本低，团队成员可快速上手
- **负面影响：** 随着代码量增长，模块边界可能逐渐模糊，需要通过代码审查和规范来约束；单点部署存在可用性风险（MVP阶段可接受）
- **需要关注的风险：** 如果用户量快速增长超过1000并发，需要提前规划数据库读写分离和热点模块拆分

## ADR-002：选择 H5 移动网页作为用户端平台

### 状态
已接受

### 背景
平台面向在校大学生，用户端需要覆盖iOS和Android。需要选择用户端技术方案：H5移动Web、微信小程序、React Native/uni-app跨平台App。

### 决策
我们决定以**Vue 3 + Vant（移动端UI库）构建 H5 移动网页**作为用户端平台，管理端使用Vue 3 + Element Plus。

### 理由
1. **技术栈统一：** 用户端和管理端均使用Vue 3，团队只需掌握一套前端框架，学习成本最低，代码复用率最高
2. **无平台审核限制：** H5网页上线无需等待微信小程序审核，迭代发布更灵活，不受平台政策变化影响
3. **用户获取门槛低：** 手机浏览器直接访问，无需安装任何App，通过链接或二维码即可分享传播
4. **开发效率高：** Vant组件库提供完善的移动端UI组件（列表、表单、弹窗、下拉刷新等），开发速度接近小程序原生
5. **PWA支持：** 通过PWA技术可将H5添加到手机桌面，体验接近原生App，弥补了传统H5"无桌面入口"的不足
6. **响应式适配：** 响应式设计可同时适配手机和PC浏览器，一套代码覆盖多端

### 后果
- **正面影响：** 技术栈与管理端统一，降低学习和维护成本；无平台审核限制，上线和迭代更灵活；跨平台兼容性好
- **负面影响：** 无法直接使用微信原生能力（如一键获取手机号、消息模板推送），需要自行实现等效功能；H5在手机浏览器中的性能和体验略逊于原生小程序
- **需要关注的风险：** 需要自行实现用户认证（手机号验证码登录替代微信授权）；消息推送需依赖WebSocket或邮件通知，无法使用微信模板消息

## ADR-003：选择MySQL + Redis作为数据存储方案

### 状态
已接受

### 背景
平台需要存储用户信息、任务数据、订单记录、评价记录、聊天消息等。需要选择合适的数据库方案。候选方案包括：MySQL单库、MySQL + Redis、MySQL + MongoDB、PostgreSQL + Redis。

### 决策
我们决定采用**MySQL 8.0 作为主数据库，Redis 7.0 作为缓存和会话存储**。

### 理由
1. **数据模型匹配：** 平台核心数据（用户、订单、任务、评价）都是结构化关系型数据，MySQL的关系模型天然匹配
2. **团队熟悉度：** 团队成员在课程中已掌握MySQL的SQL编写、索引优化、事务管理
3. **Redis缓存场景明确：**
   - 用户登录会话（JWT Token黑名单）
   - 需求大厅列表缓存（高频读取，减少MySQL压力）
   - 支付状态缓存（String，防重复支付）
   - WebSocket连接状态（用户ID → 连接映射）
4. **不选MongoDB的原因：** 虽然聊天消息适合文档存储，但MVP阶段消息量可控，MySQL JSON字段可暂代。引入MongoDB会增加运维和开发复杂度
5. **不选PostgreSQL的原因：** MySQL的社区资料和Docker镜像更丰富，在团队已有的Spring Boot学习体系中集成更顺畅

### 后果
- **正面影响：** 单一关系型数据库降低运维成本，Redis作为缓存显著提升热点数据读取性能
- **负面影响：** 聊天消息存储在MySQL中，消息量增长后可能需要迁移至MongoDB或专用IM存储；MySQL的JSON字段查询效率不如MongoDB
- **需要关注的风险：** 聊天消息表需要定期归档清理，避免单表过大影响查询性能

### AI 辅助记录

**AI 初稿（DeepSeek，角色扮演架构师）：**

> 作为架构师，我推荐采用 **MySQL + Redis + MongoDB** 的多数据库方案，针对不同数据特征选择最合适的存储引擎。
>
> **1. MySQL 8.0 — 主数据库**
>
> 存储结构化核心业务数据：
> - 用户表（user）：用户基本信息、学号认证
> - 任务表（task）：跑腿任务发布信息、状态、报酬
> - 订单表（order）：接单记录、状态流转、完成凭证
> - 评价表（rating）：交易互评、好评率
>
> 理由：核心业务数据是典型的关系型数据，有外键关联、事务需求（如支付转账需要事务保证），MySQL 的 ACID 事务支持是必须的。
>
> **2. Redis 7.0 — 缓存层**
>
> 用于热点数据缓存，减轻 MySQL 读压力。建议缓存需求大厅列表、用户会话等高频访问数据。
>
> **3. MongoDB — 文档存储**
>
> 存储非结构化/半结构化数据：
> - 聊天消息（message）：消息格式灵活（文字、图片、系统通知），MongoDB 的文档模型天然匹配
> - 评价详情（rating_detail）：评价内容可能包含图片、标签等自由格式数据
>
> 理由：聊天消息的 Schema 灵活（消息类型、附件、已读状态等字段可能随迭代变化），MongoDB 的灵活 Schema 避免了 MySQL 的频繁 ALTER TABLE。且 MongoDB 天然支持按时间范围分片，适合消息数据的水平扩展。
>
> **4. Elasticsearch — 搜索引擎（可选）**
>
> 用于需求大厅的全文搜索功能。MySQL 的 LIKE 查询在数据量增长后性能下降明显，Elasticsearch 支持分词、拼音搜索、高亮显示，可显著提升搜索体验。
>
> **总结：**
>
> | 数据类型 | 存储引擎 | 理由 |
> |---------|---------|------|
> | 用户、任务、订单、评价 | MySQL | 关系型数据，需事务支持 |
> | 会话、缓存、排行榜 | Redis | 高频读写，内存速度 |
> | 聊天消息、评价详情 | MongoDB | Schema 灵活，适合文档型数据 |
> | 全文搜索 | Elasticsearch | 分词搜索，提升搜索体验 |

**修订理由：** AI 的推荐是"教科书式"多数据库方案——每种数据选最合适的数据库。但在 4 人 10 周约束下，每多一种数据库就多一套运维/监控/学习成本。工程上优先"够用"而非"最优"，MVP 阶段 MySQL + Redis 完全能满足所有数据需求。Elasticsearch 在 200 用户场景下更是典型过度设计。

## ADR-004：选择WebSocket实现即时通讯，不引入独立IM服务

### 状态
已接受

### 背景
平台需要在任务双方之间提供文字和图片聊天功能。候选方案包括：集成第三方IM SDK（如腾讯云IM、环信）、自建WebSocket服务、使用轮询。

### 决策
我们决定使用**Spring WebSocket + STOMP协议**自建IM功能，不引入第三方IM SDK。

### 理由
1. **功能范围可控：** MVP的IM需求仅限于任务内1对1聊天（文字+图片），不涉及群聊、音视频、朋友圈等复杂功能。Spring WebSocket原生支持足够
2. **成本考量：** 第三方IM SDK（腾讯云IM、环信）的免费额度通常限制MAU在100以内，超出后需要付费。自建方案无额外费用
3. **数据可控性：** 聊天记录作为纠纷仲裁证据，需要存储在自有数据库中。自建方案对数据的完全控制比第三方SDK更方便合规
4. **技术可行性：** Spring Boot对WebSocket有原生支持（spring-boot-starter-websocket），STOMP协议简化了消息路由和订阅管理
5. **不选轮询的原因：** 轮询的实时性和资源消耗都不如WebSocket，用户体验差

### 后果
- **正面影响：** 零额外成本，聊天数据完全可控，与现有Spring Boot技术栈无缝集成
- **负面影响：** WebSocket长连接管理（断线重连、心跳检测）需要自行实现；单机部署时所有WebSocket连接在同一实例，横向扩展时需要引入消息中间件来做跨实例消息推送
- **需要关注的风险：** H5页面切后台或锁屏后WebSocket可能被浏览器回收，需实现自动重连和心跳检测；随着用户量增长，如果单机WebSocket连接数接近上限（约1万），需要提前规划集群方案
