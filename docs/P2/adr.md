## ADR-001：采用单体分层架构而非微服务架构

### 状态
已接受

### 背景
项目需要为校园互助平台选择整体架构风格。P1最新需求已将MVP范围收敛为：用户注册与学号实名认证、服务板块的代取服务、互助板块的失物招领和二手交易、即时通讯、评价与好评率、支付宝沙箱支付、后台审核与申诉处理。P2架构设计据此明确为9个单体后端内部逻辑模块：用户、代取服务、支付、失物招领、二手交易、即时通讯、评价、管理、文件存储。找搭子、咨询问答、代购/代办保留为后续规划，不进入MVP架构模块。团队4人，开发周期10周，MVP阶段目标并发量200人。

### 决策
我们决定采用**前后端分离 + 后端单体分层架构**（Controller-Service-Repository），而非微服务架构。

### 理由
1. **团队规模与生产力：** 4人团队维护多个微服务会导致大量时间花在服务间通信、配置管理、本地调试上，而非业务开发
2. **开发周期约束：** 10周内需要完成MVP，微服务的基础设施（服务发现、配置中心、链路追踪、CI/CD多服务流水线）搭建至少需要2-3周
3. **并发量不构成拆分驱动力：** 200并发完全可以通过单实例、MySQL索引和可选缓存优化解决，无需微服务的独立扩缩容能力
4. **模块化设计预留演进路径：** 分层架构内部通过良好的模块边界设计，允许未来将高频模块（如IM服务、代取服务）逐步拆分为独立服务
5. **团队技术储备：** 团队在课程中系统学习了Spring Boot分层架构，无微服务实战经验

### 后果
- **正面影响：** 开发效率高，本地调试简单，部署运维成本低，团队成员可快速上手
- **负面影响：** 随着代码量增长，模块边界可能逐渐模糊，需要通过代码审查和规范来约束；单点部署存在可用性风险（MVP阶段可接受）
- **需要关注的风险：** 如果用户量快速增长超过1000并发，需要提前规划数据库读写分离、Redis缓存和热点模块拆分

## ADR-002：选择电脑端 Web 单页应用作为前端平台

### 状态
已接受（P4回溯修订）

### 背景
P1回溯评审后，产品形态已收敛为面向电脑端浏览器的校园互助Web平台。系统需要同时支持学生用户的需求大厅、详情页、个人中心、私聊等页面，以及管理员的认证审核、内容审核、举报/申诉处理等页面。团队已确定前端基于Vue技术栈，但团队前端经验有限，需要在开发效率、学习成本和桌面端页面能力之间做取舍。

### 决策
我们决定采用**Vue 3 + Vite + JavaScript + Element Plus + Vue Router + Pinia + Axios**构建面向电脑端浏览器的**单页应用（SPA）**。学生端视图和管理端视图放在同一个Vue应用中，通过路由和角色权限区分访问范围。

### 理由
1. **匹配当前需求基线：** P1最新SRS要求系统以电脑端浏览器为主要运行环境，因此前端选型应服务于桌面端业务系统，而不是移动H5
2. **组件能力匹配：** 平台页面以表单、表格、筛选、详情、弹窗、分页、上传和审核操作为主，Element Plus比Vant等移动端组件库更适合
3. **降低团队学习成本：** 使用JavaScript而不引入TypeScript，避免额外类型系统学习和调试成本，更符合4人团队10周MVP约束
4. **统一前端应用：** 学生端和管理端共用同一个Vue应用，可复用登录态、角色权限、请求封装、基础布局和组件库
5. **前后端分离契合度高：** SPA通过Vue Router管理页面切换，后端提供RESTful API；Pinia管理跨页面状态，Axios统一封装接口请求
6. **不选其他方案的原因：** Vant面向移动端，不匹配电脑端；uni-app跨端收益有限且增加学习成本；SSR/SSG更适合SEO或静态内容场景；传统后端模板渲染会增加前后端耦合

### 后果
- **正面影响：** 技术栈清晰，桌面端组件能力充足，学生端和管理端可复用同一套登录、权限、路由和接口封装
- **负面影响：** 团队仍需编写Vue页面逻辑、接口调用和少量自定义样式；单页应用需要处理登录过期、路由守卫和刷新后状态恢复
- **需要关注的风险：** Element Plus默认风格偏后台系统，需要在页面组织上区分学生端和管理端体验；若后续重新转向移动端，需要重新评估组件库和适配方案

## ADR-003：选择MySQL作为主存储，Redis作为可选缓存优化

### 状态
已接受（P4回溯修订）

### 背景
平台需要存储用户信息、代取服务数据、互助发布内容、评价记录、聊天消息、支付宝沙箱交易记录等。P1需求没有强制规定具体数据存储技术，P2需要根据团队技术储备和10周MVP约束确定后端数据方案。P4回溯后，有报酬代取服务采用待支付锁单流程：服务记录和支付记录先以待支付状态保存，发布者可继续支付或取消，支付超时由系统自动取消，支付成功后再进入内容审核。候选方案包括：MySQL单库、MySQL + Redis、MySQL + MongoDB、PostgreSQL + Redis。

### 决策
我们决定采用**MySQL 8.0作为MVP必选主数据库，MyBatis Plus作为数据访问框架**。Redis 7.0不作为MVP必选组件，仅作为后续可选缓存优化；如果后期进度允许或压测发现性能不足，再引入Redis。

### 理由
1. **数据模型匹配：** 平台核心数据（用户、代取服务、互助发布内容、评价）都是结构化关系型数据，MySQL的关系模型天然匹配
2. **团队熟悉度：** 团队成员在课程中已掌握MySQL的SQL编写、索引优化、事务管理；团队没有Spring Data JPA实践经验，MyBatis Plus更贴近SQL使用习惯，能降低常规CRUD和分页查询开发成本
3. **MVP组件最小化：** Redis虽适合缓存和临时状态，但不是P1需求的必要实现条件，且团队目前没有Redis实践经验。待支付锁单可先通过MySQL中的服务状态、支付状态和过期时间字段实现。MVP阶段可先通过MySQL索引、分页查询、事务控制、唯一约束和单体应用内状态管理满足需求
4. **Redis作为后续优化：** 若后续有时间或压测发现瓶颈，再用Redis实现需求大厅热点列表缓存、Token临时状态、登录限流或后续图形验证码状态、支付宝沙箱防重复支付/退款、WebSocket在线状态管理
5. **不选MongoDB的原因：** 虽然聊天消息适合文档存储，但MVP阶段消息量可控，使用MySQL表结构或JSON字段即可。引入MongoDB会增加运维和开发复杂度
6. **不选PostgreSQL的原因：** MySQL的社区资料和Docker镜像更丰富，在团队已有的Spring Boot学习体系中集成更顺畅

### 后果
- **正面影响：** MVP必选存储组件收敛为MySQL，部署和调试成本最低；MyBatis Plus贴近团队已有SQL经验，提升常规数据访问开发效率
- **负面影响：** 没有Redis时，热点列表缓存、临时状态和在线状态管理需要先用MySQL或应用内状态实现，性能和横向扩展能力有限
- **需要关注的风险：** 聊天消息表需要定期归档清理，避免单表过大影响查询性能；若压测发现需求大厅或在线状态成为瓶颈，需要及时引入Redis

### AI 辅助记录

**AI 初稿（DeepSeek，角色扮演架构师）：**

> 作为架构师，我推荐采用 **MySQL + Redis + MongoDB** 的多数据库方案，针对不同数据特征选择最合适的存储引擎。
>
> **1. MySQL 8.0 — 主数据库**
>
> 存储结构化核心业务数据：
> - 用户表（user）：用户基本信息、学号认证
> - 任务表（task）：跑腿任务发布信息、状态、报酬
> - 订单表（order）：接单记录、状态流转、完成凭证
> - 评价表（rating）：交易互评、好评率
>
> 理由：核心业务数据是典型的关系型数据，有外键关联、事务需求（如支付宝沙箱转账和退款状态记录需要事务保证），MySQL 的 ACID 事务支持是必须的。
>
> **2. Redis 7.0 — 缓存层**
>
> 用于热点数据缓存，减轻 MySQL 读压力。建议缓存需求大厅列表、用户会话等高频访问数据。
>
> **3. MongoDB — 文档存储**
>
> 存储非结构化/半结构化数据：
> - 聊天消息（message）：消息格式灵活（文字、图片、系统通知），MongoDB 的文档模型天然匹配
> - 评价详情（rating_detail）：评价内容可能包含图片、标签等自由格式数据
>
> 理由：聊天消息的 Schema 灵活（消息类型、附件、已读状态等字段可能随迭代变化），MongoDB 的灵活 Schema 避免了 MySQL 的频繁 ALTER TABLE。且 MongoDB 天然支持按时间范围分片，适合消息数据的水平扩展。
>
> **4. Elasticsearch — 搜索引擎（可选）**
>
> 用于需求大厅的全文搜索功能。MySQL 的 LIKE 查询在数据量增长后性能下降明显，Elasticsearch 支持分词、拼音搜索、高亮显示，可显著提升搜索体验。
>
> **总结：**
>
> | 数据类型 | 存储引擎 | 理由 |
> |---------|---------|------|
> | 用户、任务、订单、评价 | MySQL | 关系型数据，需事务支持 |
> | 会话、缓存、排行榜 | Redis | 高频读写，内存速度 |
> | 聊天消息、评价详情 | MongoDB | Schema 灵活，适合文档型数据 |
> | 全文搜索 | Elasticsearch | 分词搜索，提升搜索体验 |

```diff
+ 虽然聊天消息适合文档存储，但 MVP 阶段消息量可控，MySQL JSON 字段可暂代。
+ 引入 MongoDB 会增加运维和开发复杂度（多一套数据库的备份、监控、学习成本）。

- Elasticsearch — 搜索引擎：
- 用于需求大厅全文搜索，支持分词、拼音搜索、高亮显示。
+ 不选 Elasticsearch 的原因：
+ 200 用户、数十条需求的场景下，MySQL 索引完全够用。
+ Elasticsearch 是典型过度设计——多一套中间件就多一份运维风险。

+ 风险说明：聊天消息量增长后可能需迁移至 MongoDB；需定期归档清理避免单表过大。
```

**修订理由：** AI 的推荐是"教科书式"多数据库方案——每种数据选最合适的数据库。但在 4 人 10 周约束下，每多一种数据库就多一套运维/监控/学习成本。P4回溯后团队进一步确认 Redis 不作为必选技术，MVP 阶段优先保证 Spring Boot + MyBatis Plus + MySQL 主链路可交付；Redis 仅在后期进度允许或压测发现瓶颈时引入。Elasticsearch 在 200 用户场景下更是典型过度设计。

## ADR-004：选择WebSocket实现即时通讯，不引入独立IM服务

### 状态
已接受

### 背景
平台需要在代取服务双方、失物招领和二手交易的内容发布者与联系人之间提供一对一文字/图片聊天功能。候选方案包括：集成第三方IM SDK（如腾讯云IM、环信）、自建WebSocket服务、使用轮询。

### 决策
我们决定使用**Spring WebSocket**实现站内一对一文字/图片消息推送。图片本体通过本地文件存储保存，聊天消息记录保存图片文件标识和元数据。聊天入口由业务状态控制：代取服务仅在进行中状态允许双方私聊；失物招领和二手交易仅在内容已发布状态允许联系发布者。

### 理由
1. **功能范围可控：** MVP的IM需求仅限于代取服务、失物招领和二手交易场景内的1对1文字/图片聊天，不涉及群聊、音视频、朋友圈等复杂功能。Spring WebSocket原生支持足够
2. **成本考量：** 第三方IM SDK（腾讯云IM、环信）的免费额度通常限制MAU在100以内，超出后需要付费。自建方案无额外费用
3. **数据可控性：** 聊天记录作为纠纷或申诉处理证据，需要存储在自有数据库中。自建方案对数据的完全控制比第三方SDK更方便合规
4. **技术可行性：** Spring Boot对WebSocket有原生支持，前端可使用浏览器原生WebSocket API完成连接、发送和接收
5. **不选轮询的原因：** 轮询的实时性和资源消耗都不如WebSocket，用户体验差

### 后果
- **正面影响：** 零额外成本，聊天数据完全可控，与现有Spring Boot技术栈无缝集成
- **负面影响：** WebSocket长连接管理（断线重连、心跳检测）需要自行实现；单机部署时所有WebSocket连接在同一实例，横向扩展时需要引入消息中间件来做跨实例消息推送
- **需要关注的风险：** 电脑端浏览器可能因断网、刷新、标签页休眠或多标签登录导致WebSocket连接中断或状态不一致，需实现自动重连、心跳检测和会话状态恢复；随着用户量增长，如果单机WebSocket连接数接近上限（约1万），需要提前规划集群方案
