## ADR-001：采用单体分层架构而非微服务架构

### 状态
已接受

### 背景

项目需要为校园互助服务平台选择整体架构风格。P1 v2.1 已将 MVP Must 范围收缩为“可信代取服务闭环”：用户注册登录与个人资料、实名认证提交与管理员审核、代取需求大厅、代取发布、待支付锁单、接单、完成凭证、确认完成、取消/超时取消、支付宝沙箱预付款/退款/结算、我的代取记录。评价体系和站内通知为 Should；失物招领、二手交易、私聊、举报/申诉为 Could；真实线上支付、论坛、找搭子、学习辅导、智能推荐、微服务和事件驱动/MQ 为 Won't。

团队 4 人，开发周期 10 周，目标是形成可运行、可演示、可测试的 MVP，而不是验证分布式架构能力。

### 决策

采用**前后端分离 + 后端单体分层架构**（Controller-Service-Mapper/Repository），不采用微服务架构。

### 理由

1. **业务域集中：** 最新 Must 需求都围绕代取服务闭环，不存在必须按业务板块拆分服务的需求。
2. **开发周期约束：** 10 周内需要完成实现、联调、测试和演示，微服务基础设施会挤占核心业务开发时间。
3. **一致性更易控制：** 待支付、支付成功、待接单、接单、完成、退款和结算等状态流转在单体内同步调用更容易保证一致性。支付记录中的业务追踪字段仅用于追踪、排错和幂等校验；代取服务状态仍由代取服务模块统一维护。
4. **团队学习成本低：** Spring Boot 分层架构和 MySQL 是团队已有技术基础，微服务涉及服务治理、分布式事务和多服务 CI/CD，风险更高。
5. **可演进：** 评价、通知、失物招领、二手交易、私聊等可先作为单体内部逻辑模块或后续扩展模块，未来有真实规模压力时再拆分。

### 后果

- **正面影响：** 开发效率高，本地调试简单，部署链路短，团队能把精力放在代取核心流程和测试上。
- **负面影响：** 代码量增长后模块边界可能模糊，需要通过包结构、接口约定和代码评审约束。
- **需要关注的风险：** 若后续真实用户量增长、通知/私聊成为高频能力，可能需要补充 Redis、异步任务或拆分热点模块。

### AI 辅助记录

**AI 初稿摘要：**
AI 在提供“4 人、10 周、MVP、初期并发约 200 人”等约束后，推荐前后端分离 + 后端单体分层架构，并指出微服务会增加开发和运维成本。

**人工修订内容：**

```diff
  推荐前后端分离 + 后端单体分层架构。
- MVP 功能包括跑腿代取、失物招领、二手交易、即时通讯、后台管理等。
+ MVP Must 范围收缩为用户管理、实名认证审核、代取需求大厅、代取闭环、我的代取记录。
+ 评价和通知为 Should；失物招领、二手交易、私聊、举报/申诉为 Could。
+ 微服务、事件驱动/MQ、智能推荐等为 Won't。

- 技术栈可使用 Spring Boot + MySQL + Redis + WebSocket。
+ MVP 必选链路为 Spring Boot + MySQL + 本地文件存储 + 支付宝沙箱。
+ Redis、WebSocket、MQ 均不作为 MVP 必选组件。

+ 补充待支付、退款、结算等代取状态流转在单体内更易保证一致性的理由。
+ 补充“10 周内可运行、可演示、可测试”作为关键判断标准。
```

**修订理由：** AI 的单体分层结论可采纳，但其输入基线包含旧的泛互助平台范围，且仍倾向附带 Redis、WebSocket 等组件。P1 v2.1 进一步收缩后，架构决策必须围绕代取闭环重新描述，避免让可选功能反向抬高 MVP 复杂度。

## ADR-002：选择电脑端 Web 单页应用作为前端平台

### 状态
已接受

### 背景

P1 v2.1 明确产品是面向电脑端浏览器的校园互助 Web 平台。学生端需要代取大厅、代取详情、发布代取、我的发布、我的接单、评价和通知等页面；管理端需要实名认证审核页面。团队前端经验有限，需要在开发效率、组件成熟度和桌面端业务页面能力之间做取舍。

### 决策

采用 **Vue 3 + Vite + JavaScript + Element Plus + Vue Router + Pinia + Axios** 构建电脑端 Web 单页应用（SPA）。学生端视图和管理端视图放在同一个 Vue 应用中，通过路由和角色权限区分访问范围。

### 理由

1. **匹配运行环境：** P1 要求电脑端浏览器，不需要移动 H5、小程序或跨端框架。
2. **组件能力匹配：** 代取发布、筛选、列表、详情、文件上传、认证审核等都需要桌面端表单和表格能力，Element Plus 更适合。
3. **降低学习成本：** 使用 JavaScript 而非 TypeScript，减少团队在 10 周内的额外学习成本。
4. **统一应用：** 学生端和管理端共用登录态、路由守卫、角色权限、请求封装和基础组件。
5. **前后端分离：** SPA 通过 Axios 调用 REST API，后端无需渲染页面，职责边界清晰。

### 后果

- **正面影响：** 页面开发效率高，桌面端组件成熟，学生端和管理端可以复用工程配置和基础组件。
- **负面影响：** Element Plus 默认风格偏后台系统，学生端页面需要适当调整布局和视觉层级，避免体验过于管理后台化。
- **需要关注的风险：** SPA 需要处理刷新后的登录态恢复、路由守卫、Token 过期和接口错误提示。

### AI 辅助记录

**AI 初稿摘要：**
P2 原始阶段 AI 基于“H5 用户端 + Web 管理端”的旧约束，推荐 Vue 3 + uni-app/Vant 作为用户端，Vue 3 + Element Plus 作为管理端。

**人工修订内容：**

```diff
- 用户端方案：H5 移动网页，Vue 3 + Vant 或 uni-app。
- 管理端方案：Vue 3 + Element Plus。
+ 前端平台：电脑端 Web 单页应用。
+ 技术栈：Vue 3 + Vite + JavaScript + Element Plus + Vue Router + Pinia + Axios。
+ 学生端和管理端放在同一个 Vue 应用中，通过路由和角色权限区分。

- 选择 Vant/uni-app 是为了移动端适配和未来跨端。
+ 不选 Vant/uni-app，因为 P1 v2.1 已明确电脑端浏览器为主要环境，跨端能力收益不足。
+ 不引入 TypeScript，优先降低 4 人团队 10 周 MVP 的实现风险。
```

**修订理由：** AI 的原始建议受旧需求影响，前端平台判断已经不适用。当前架构应服务于电脑端业务页面和演示交付，Element Plus + Vue SPA 是更直接的方案。

## ADR-003：选择 MySQL 作为主存储，Redis 作为可选缓存优化

### 状态
已接受

### 背景

平台需要存储用户、实名认证申请、代取服务、支付沙箱交易记录、文件元数据、评价记录和通知记录。P1 v2.1 的数据主要是结构化关系型数据，并且核心业务需要围绕代取状态流转、学号唯一性、支付记录和评价记录进行查询与约束。候选方案包括 MySQL 单库、MySQL + Redis、MySQL + MongoDB、PostgreSQL + Redis。

### 决策

采用 **MySQL 8.0 作为 MVP 必选主数据库，MyBatis Plus 作为数据访问框架**。Redis 7.0 不作为 MVP 必选组件，仅作为后续可选缓存优化；MVP 阶段不引入 MongoDB、Elasticsearch 或其他数据库。

### 理由

1. **数据模型匹配：** 用户、实名认证、代取服务、支付记录、评价和通知都是结构化数据，MySQL 关系模型适配度高。
2. **约束能力需要：** 学号唯一性、用户名唯一性、代取状态、支付状态和评价资格校验都适合通过 MySQL 约束、索引和事务配合实现。
3. **团队熟悉度：** 团队已有 MySQL 和 SQL 基础，MyBatis Plus 能降低常规 CRUD、分页和条件查询工作量。
4. **MVP 组件最小化：** Redis 可用于热点列表缓存、防重复操作或登录限流，但不是 P1 Must 需求的必要条件。
5. **不选多数据库：** MongoDB、Elasticsearch 对当前数据量和功能范围收益有限，会增加部署、备份和学习成本。

### 后果

- **正面影响：** 数据链路简单，部署和调试成本低，便于快速完成核心业务和测试。
- **负面影响：** 没有 Redis 时，热点大厅列表只能先依赖 MySQL、应用内逻辑和分页索引优化，性能弹性有限。
- **需要关注的风险：** MVP 阶段采用定时任务扫描 + 业务访问时惰性校验结合处理待支付和待接单超时取消；Redis 不作为该机制的必选组件。如果后续压测发现列表查询慢，应及时补充索引或引入 Redis。

### AI 辅助记录

**AI 初稿摘要：**
AI 曾推荐 MySQL + Redis + MongoDB + Elasticsearch 的多数据库方案，理由是不同数据类型适合不同存储引擎。

**人工修订内容：**

```diff
- 推荐 MySQL + Redis + MongoDB，并可选 Elasticsearch。
+ 推荐 MySQL 8.0 作为 MVP 必选主数据库，MyBatis Plus 作为数据访问框架。
+ Redis 仅作为后续可选缓存优化。
+ 不引入 MongoDB 和 Elasticsearch。

- MongoDB 用于聊天消息，Elasticsearch 用于全文搜索。
+ 私聊为 Could，关键词搜索为 Could，不应成为 MVP 存储架构驱动力。
+ 当前核心数据为用户、实名认证、代取服务、支付记录、评价和通知，MySQL 足够支撑。
```

**修订理由：** AI 的多数据库建议是“理论最优”，但与 4 人 10 周 MVP 不匹配。每多一种数据库就多一套部署、学习、备份和排错成本。当前应优先保证 MySQL 主链路可靠。

## ADR-004：站内通知采用数据库通知记录，不引入 WebSocket 或消息队列

### 状态
已接受（P1 v2.1 回溯修订）

### 背景

P1 v2.1 将站内通知设为 Should 需求，用于展示认证审核结果、接单、完成凭证上传、确认完成和收到评价等事件，并支持已读/未读状态。私聊与聊天记录已降为 Could，事件驱动和 MQ 为 Won't。候选方案包括：数据库通知记录、WebSocket 实时推送、消息队列异步通知、第三方消息服务。

### 决策

MVP 阶段站内通知采用**数据库通知记录 + 前端主动刷新 + 30 秒低频轮询未读数量**方案。业务模块在关键事件发生时同步创建通知记录；前端在登录成功后、进入首页、进入通知页、完成关键业务操作后主动查询未读通知数量，用户保持登录状态时每 30 秒查询一次未读通知数量。通知列表仅在进入通知页或手动刷新时查询。不引入 WebSocket、RabbitMQ/Kafka 或第三方消息服务。

### 理由

1. **匹配优先级：** 通知是 Should，不是阻断代取闭环的 Must；不应引入重型实时通信架构。
2. **功能足够：** P1 只要求通知记录和已读/未读状态，没有要求秒级推送、离线推送或多端消息同步。
3. **实现简单：** 数据库通知记录即可保存必要通知信息和已读/未读状态。
4. **降低运维成本：** 不引入 WebSocket 长连接和 MQ，减少部署、心跳、断线重连、消息积压和消费失败等问题。
5. **边界清晰：** 后续如果私聊或实时提醒升级为 Must，应单独重新评估通信方案，不影响当前 MVP 通知记录方案。

### 后果

- **正面影响：** 实现成本低，测试简单，通知可追溯，符合 MVP 交付节奏。
- **负面影响：** 用户不能立即收到实时推送，需要刷新、进入通知页或依赖 30 秒低频轮询未读数量。
- **需要关注的风险：** 前端只轮询未读数量，通知列表按需查询，避免持续拉取列表造成不必要请求。

### AI 辅助记录

**AI 初稿摘要：**
AI 在旧需求基线下曾建议使用 Spring WebSocket 实现即时通讯，甚至比较过第三方 IM SDK、自建 WebSocket 和轮询方案。

**人工修订内容：**

```diff
- 决策：使用 Spring WebSocket 实现站内一对一文字/图片消息推送。
- 背景：平台需要支持代取、失物招领、二手交易场景内私聊。
+ 决策：站内通知采用数据库通知记录 + 前端主动刷新 + 30 秒低频轮询未读数量。
+ 背景：P1 v2.1 中通知为 Should，私聊为 Could，事件驱动和 MQ 为 Won't。

- WebSocket 用于聊天实时性和图片消息。
+ MVP 不实现私聊，不需要 WebSocket 长连接。
+ 通知只需记录认证结果、接单、完成凭证、确认完成和收到评价等事件。

+ 通知列表仅在进入通知页或手动刷新时查询，不引入 WebSocket、RabbitMQ/Kafka 或第三方消息服务。
```

**修订理由：** 原 WebSocket 决策依赖旧的私聊需求。最新 P1 已将私聊降为 Could，通知也只是 Should，因此更合理的架构选择是保留可追溯的通知记录，不为非核心能力引入长连接和消息中间件。
